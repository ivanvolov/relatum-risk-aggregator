# Architecture

Internal map of this repo. **Not** public-facing — that's `README.md`.

If you're new here, read this in order:
1. `RFP.md` — what we're building and why (verbatim EF brief). Source of truth for scope, the no-composite-scoring rule, the protocol+feed list.
2. This file — how the code is laid out.
3. `PROPOSAL.md` — our application reply (only relevant when editing the proposal itself; the code doesn't depend on it).

---

## What this repo is

A React SPA + a build pipeline that turns RDF/Turtle claims into a coverage matrix and per-protocol detail pages for the [EF Neutral DeFi Risk Intelligence Aggregator RFP](./RFP.md).

The structural rule: **no composite scoring, ever**. Every cell on the matrix is one feed's verbatim claim about one protocol, with provenance. This is enforced at the schema layer — the RDF vocabulary has no predicate for "Relatum's score." See `data/ontology/vocab.ttl`.

The deploy is GH Pages → `https://ivanvolov.github.io/relatum-risk-aggregator/`. Static SPA; the data layer is JSON files fetched at runtime from the same origin.

## §1 — Layering & source-of-truth direction

Data flows in **one direction**, always:

```
   (1) feed APIs            (2) RDF graph              (3) JSON               (4) React
   (operator runs           (committed, the            (build artifact         (browser
   adapter.mjs)             canonical store)           in public/data/)        fetches)
        ↓                          ↓                        ↓                       ↓
   data/adapters/<feed>/    data/ontology/           public/data/            src/lib/data.js
   output/<date>.json       protocols/<slug>/        protocols/<slug>.json     ↓
                            <feed>.<date>.ttl                                src/routes/*
```

**The committed source of truth is the TTL.** The per-feed scrape JSONs in `data/adapters/<feed>/output/*.json` are committed too (so the pipeline is fully re-derivable without re-running scrapers), but the canonical claims live in `data/ontology/protocols/<slug>/<feed>.<date>.ttl`.

**`public/data/` is a build artifact.** It's committed for GH Pages convenience (Actions runs `vite build`, not the data pipeline — see §6), but never hand-edit it; it's overwritten by `npm run build:data`.

`vite.config.js` sets `publicDir: 'public'` so Vite serves `public/data/index.json` as `/data/index.json` in dev, and copies the same files into `dist/` on build.

## §3 — `data/` (everything data-related lives here)

This is the answer to "what the fuck is built, ontology, seeds, legacy." All four are siblings inside `data/`; each has one job.

```
data/
├── ontology/             ← (a) RDF schema + the actual claim graph
├── adapters/             ← (b) one folder per risk feed: scraper + schema + RDF mapper + outputs
├── seeds/                ← (c) hand-curated YAML fallbacks (when no TTL exists yet)
└── build/                ← (d) Node scripts: TTL → JSON → index. Runs `npm run build:data`.
```

### (a) `data/ontology/` — the RDF graph (canonical store)

```
data/ontology/
├── vocab.ttl                 the rfp: schema. Classes (Subject, Feed, Claim, Rating, Score,
│                             Finding, CoverageStatement) and predicates (statedBy, sourceURL,
│                             observedAt, dimension, ratingValue, etc.). EDIT WITH CARE — a
│                             schema change ripples through every consumer.
├── feeds.ttl                 rfp:Feed instances (closed set of 15 feeds). One block per feed
│                             with rdfs:label + rfp:methodologyURL.
├── _methodology.md           prose notes (not parsed)
└── protocols/
    └── <slug>/               one folder per ontology slug (NOT the frontend slug — see §5)
        ├── <feed>.<date>.ttl   per-feed per-snapshot claim files (adapter-emitted, dated)
        ├── <feed>.ttl          per-feed hand-curated claim files (only aave-v3 has these)
        ├── combined.ttl        per-protocol merge of all per-feed TTLs (committed)
        └── _prose.yaml         optional hand-curated narrative overlay (sidecards, audits,
                                incidents, per-feed methodology blurbs). Currently only aave-v3.
```

**Why dated filenames** (`defiscan.2026-06-17.ttl`): every claim in the file carries `rfp:observedAt = <that date>`. Re-running the adapter on a new date writes a *new* file alongside the old one, so history accumulates as additional triples rather than overwriting. `combine-ttl.mjs` unions all per-feed TTLs; the history split happens later in `ttl-to-json.mjs` via `splitCurrentVsHistory()` (see `data/build/lib/ttl-load.mjs`).

**Why aave-v3 is special.** It has hand-curated `<feed>.ttl` (undated) for 8 of the 15 feeds (`blockanalitica`, `credora`, `curatorwatch`, `defisaver`, `defiscan`, `defisphere`, `pharos`, `stablewatch`), plus `_prose.yaml`. That's our "worked example" — it shows what a fully-mapped protocol looks like for the feeds we've hand-curated so far. The remaining 7 feeds (`defipunkd`, `llamarisk`, `philidor`, `pigi`, `risklayer`, `xerberus`, `zyfai`) will get aave-v3 TTLs as their adapters mature. The other 19 protocols are mostly fed by the adapter pipeline; they grow toward the aave-v3 shape as adapters land.

**Slug mapping** (`aave` frontend → `aave-v3` ontology dir): see `data/build/lib/registry.mjs:slugToOntologyDir()`. Set `ontologyDir: 'aave-v3'` in the `PROTOCOLS` registry to map a frontend slug to a different ontology folder. Only needed when versioning matters.

### (b) `data/adapters/` — one folder per risk feed

```
data/adapters/<feed>/
├── adapter.mjs              scraper. Reads external API, writes output/<YYYY-MM-DD>.json.
│                            Operator-run (no schedule yet; W3–W9 deliverable per the RFP).
│                            Present in every adapter folder.
├── schema.yaml              declared output shape — both documentation AND validation.
│                            Surfaced in the UI feed registry as a "view schema" link.
│                            Present in 12 of 15 adapter folders today
│                            (missing: llamarisk, pigi, zyfai).
├── to-ttl.mjs               RDF mapper. Reads every output/*.json, emits per-protocol-per-date
│                            TTL fragments under data/ontology/protocols/<slug>/<feed>.<date>.ttl.
│                            **Presence of this file is what flips adapterStatus to "implemented".**
│                            Only `defiscan` has one today.
├── output/<YYYY-MM-DD>.json dated raw snapshots (committed; small). Every adapter folder
│                            has at least one.
├── README.md                feed-specific notes: source URL, auth requirements, gotchas.
│                            Convention, not enforced — only `defiscan` has one today.
└── _legacy/                 see below
```

**Adapter status — the most important UX bit.** The pipeline computes `adapterStatus: 'implemented' | 'pending'` per feed by checking `existsSync(data/adapters/<feed>/to-ttl.mjs)`. This drives the matrix dot color (grey/hatched for pending, green/orange/red for implemented) AND the per-protocol-page card variant (`PendingAdapterCard` for pending, `FeedCard` for implemented). It's the difference between *"this feed exists but we haven't wired up the mapper"* and *"this feed doesn't cover this protocol."*

To promote a feed from pending → implemented:
1. Write the scraper (`adapter.mjs`) → commit a fresh `output/<date>.json`
2. Write the RDF mapper (`to-ttl.mjs`) — this is what the pipeline keys on
3. Run `node data/adapters/<feed>/to-ttl.mjs` to emit per-protocol TTL fragments
4. Run `npm run build:data` to regenerate JSON

**Current status:** 15 adapter folders exist (one per RFP feed). Only `defiscan` has `to-ttl.mjs` so far. The other 14 have `adapter.mjs` + `schema.yaml` + `output/2026-06-17.json` but no mapper → they render as "adapter pending" everywhere.

**`data/adapters/_legacy/`** is the W3–W9 one-shot fetch runners (12 scripts that hit feed homepages via `WebFetch` and dumped raw output on 2026-06-16) plus a CHANGELOG explaining what was tried and what got gated by SPA-rendering, Cloudflare, or auth walls. It's not part of the build — `data/build/` doesn't read it. Kept because (i) the CHANGELOG documents which feeds need browser sessions vs API keys vs Playwright, and (ii) the runners are the seed of the next-gen adapters. Delete when every feed has a proper `to-ttl.mjs`.

### (c) `data/seeds/` — hand-curated YAML fallbacks

```
data/seeds/
└── coverage-seed.yaml    fallback coverage matrix. Used by ttl-to-json.mjs ONLY when neither
                          the TTL nor the prose overlay sets a feed's status. This is how the
                          summary matrix shows green/orange dots for cells we haven't mapped to
                          TTL yet but know from the W3-W9 one-shot grab. As each per-feed adapter
                          lands (public/data/feeds/<feed>.json), that feed's column here is
                          ignored in favor of the live adapter output.
```

The protocol list and feed registry live in `data/build/lib/registry.mjs` (`PROTOCOLS` + `FEED_REGISTRY`) — single source of truth. To add a protocol or feed, edit there and re-run `npm run build:data`.

Status precedence inside `ttl-to-json.mjs` (highest → lowest):
1. `_prose.yaml` explicit `status:` field
2. TTL `rfp:CoverageStatement` triple
3. Inferred from claim presence (any claims → `cov`)
4. `coverage-seed.yaml` row
5. Default `'none'`

### (d) `data/build/` — the pipeline

```
data/build/
├── package.json              n3@^2 + yaml@^2
├── build-all.mjs             orchestrator: combine → ttl-to-json → build-index
├── combine-ttl.mjs           for each <slug>/, concat <feed>.*.ttl into combined.ttl
│                             (skips if combined.ttl exists; --force to regenerate)
├── ttl-to-json.mjs           the heavy lifter. For every protocol in registry.mjs:
│                              - parse combined.ttl with n3
│                              - apply _prose.yaml overlay
│                              - apply coverage-seed.yaml fallback
│                              - split claims into current[] + history[] via observedAt
│                              - emit public/data/protocols/<slug>.json
├── build-index.mjs           emit public/data/index.json (feeds + protocols + coverage matrix)
└── lib/
    ├── registry.mjs          THE source of truth for FEED_REGISTRY (15) and PROTOCOLS (20).
    │                         Also slugToOntologyDir(). Edit this when adding a protocol/feed.
    ├── ns.mjs                rfp: URI constants + STATUS_FROM_TTL map
    └── ttl-load.mjs          n3 wrapper: loadStore, claimsByFeed, splitCurrentVsHistory
```

The pipeline is fast (<2s for all 20 protocols). Re-run is byte-identical modulo `generatedAt` in index.json.

## §4 — The five stages, end to end

| # | Stage | Reads | Writes | When it runs |
|---|---|---|---|---|
| 1 | Scrape | external feed API | `data/adapters/<feed>/output/<date>.json` | operator-triggered, by hand |
| 2 | RDF map | latest `output/<date>.json` | `data/ontology/protocols/<slug>/<feed>.<date>.ttl` | `node data/adapters/<feed>/to-ttl.mjs` |
| 3 | Combine | all `<feed>.*.ttl` per protocol | `combined.ttl` per protocol | `node data/build/combine-ttl.mjs` |
| 4 | JSON emit | `combined.ttl` + `_prose.yaml` + `coverage-seed.yaml` | `public/data/protocols/<slug>.json` × 20 | `node data/build/ttl-to-json.mjs` |
| 5 | Index | per-protocol JSON | `public/data/index.json` | `node data/build/build-index.mjs` |

Stages 3–5 chain via `data/build/build-all.mjs` (= `npm run build:data`).

Stage 1 is operator-driven by design (per project conventions: scrapers are run by hand and the output JSON committed). Stage 2 is per-adapter; running all of them at once is a TODO.

| I want to… | Edit |

## §6 — Deploy

`.github/workflows/deploy.yml` runs on push to `main`:

```
checkout → setup-node 22 → npm ci → npm run build → upload-pages-artifact dist/ → deploy-pages
```

**Note: the workflow does NOT run `npm run build:data`.** It only does `vite build`. That means whatever's in `public/data/*` at commit-time is what ships. **Commit `public/data/` after running `npm run build:data` locally.** This was a deliberate choice — the build pipeline depends on adapters that may need outbound HTTP/auth/Playwright, none of which we want to wire into a public Action yet.

## §7 — Conventions worth knowing

- **HashRouter URLs** (`/#/protocol/aave`). Ugly but Just Works on Pages with no SPA-fallback config. Revisit if we want shareable canonical URLs.
- **`data/build/` has its own `package.json`.** It depends on `n3` + `yaml`. The root `package.json` is the SPA's only (React + Vite). Keeping them split means the SPA bundle never ships with the RDF parser.
- **Every claim has `rfp:statedBy + rfp:sourceURL + rfp:observedAt`.** Triple-level provenance. If a triple lacks any of these, the UI will render incomplete or wrong. Audit by running the build and looking at the `pending`/`claims=0` columns in stdout.
- **`covered: false` in the scrape JSON** maps to a single `rfp:CoverageStatement notCovered` triple (with a reason if available). This is what makes "not yet covered" a typed observation rather than a missing row.
- **No composite scoring** is the project charter. The rfp: vocabulary literally has no predicate for "Relatum's score." Adding one would require an RFP-amendment-level decision. See `RFP.md §1, Design Principle on Scoring`.
