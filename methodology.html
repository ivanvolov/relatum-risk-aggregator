Methodology

Relatum aggregates what every major DeFi risk feed says about a protocol, side by side, without synthesis. The data substrate is an RDF knowledge graph; every UI cell traces back to a single feed and a single source URL, with provenance preserved at the triple level.

The Oracle Analogy

No single price feed is canonical. A well-designed oracle aggregates many feeds to produce one trusted value, with no individual source treated as authoritative. The aggregation is the value.

The same principle applies to DeFi risk. Relatum is the aggregation layer that doesn't exist yet — a neutral place to see what every risk feed says about a protocol, side by side, with sources intact.

✕

What Relatum does not do

  • Relatum does not assign its own risk scores.
  • Relatum does not weight or rank risk feeds against each other.
  • Relatum does not tell you whether a protocol is safe or unsafe.
  • Relatum does not endorse any protocol listed.
Committed as a project charter at v0.1. Amendment requires written EF agreement, encoded in CHARTER.md.
✓

What Relatum does do

  • Aggregates risk feed coverage for the top 20 Ethereum DeFi protocols in one place.
  • Surfaces governance data from onchain sources (Safe API) and verifiable registries.
  • Tracks which feeds have — and have not — assessed each protocol.
  • Links directly to source assessments. Ratings are shown verbatim, never normalized.
  • Maintains an open, community-correctable data registry on GitHub.

Coverage gaps as data

A protocol that has not been assessed by any risk feed is itself a signal. The absence of coverage is information. A newer protocol with 2 feeds covering it is a meaningfully different risk profile from an established one with 15. Relatum makes these gaps visible rather than hiding them behind a composite number.

In our data model, absence is a typed observation, not a missing row. Every (protocol × feed) cell carries one of three states:

covered the feed publishes an assessment of this exact subject; data is in the graph
partial the feed covers a related entity (e.g. Pharos on Aave via GHO); the relationship is modeled as an explicit graph edge
not yet explicit statement of absence, with reason. Not a TODO — a typed rfp:CoverageStatement triple.

The data substrate — RDF knowledge graph

our extension beyond the RFP brief

Every feed assessment is stored as RDF triples. Each provider's methodology, units, and scale are preserved as native graph properties. Nothing is normalized or coerced onto a common scale. The aggregator's read API is "give every claim about subject X" — the JOIN happens at query time, never at write time.

No step in this pipeline normalizes, weights, or averages. The graph stores claims; the UI displays claims. The composite layer that other aggregators would add — Relatum does not have one.

The disagreement detector — a single SPARQL query is the product

PREFIX rfp: <https://relatum.xyz/ns/rfp#>

SELECT ?dim
       (GROUP_CONCAT(CONCAT(STR(?feed),"=",?val); separator=" | ") AS ?conflicts)
WHERE {
  ?c rfp:about       ?subject ;
     rfp:statedBy    ?feed ;
     rfp:dimension   ?dim ;
     rfp:ratingValue ?val .
}
GROUP BY ?dim
HAVING (COUNT(DISTINCT ?val) > 1)

Scale headroom — why infrastructure isn't the risk

Density projections from the v0.1 graph, extrapolated to the full seed-20 protocol set and half-year of weekly snapshots. Numbers are measured for the current row; multiplied for the projected ones.

ConfigurationPer protocol× 20 protocols+ 26 weekly snapshots (half-year)

With DeFiScan permission-scanner expansion ~2.5k ~50k ~1.3M
With BlockAnalitica per-market live snapshots ~3.0k ~60k ~1.6M

The v1 graph runs in-process on a laptop. Production triplestores — Apache Jena, Blazegraph, oxigraph — handle 10⁹+ triples on a single server, so at half-year horizon we're 3–4 orders of magnitude below saturation. Infrastructure is not the risk; curation, provenance, and feed onboarding are.

Why an RDF substrate is the structural enforcement of no composite scoring: the data model has no predicate for "Relatum's score." A composite would require introducing one, which requires charter amendment, which requires written EF agreement. Neutrality is enforced at the schema layer, not at the UI layer.

Data provenance

Every data point on Relatum carries a source tag. Provenance is rendered next to the value on every page; auditable to its origin URL at query time.

[onchain] fetched directly from chain or a verified onchain source (Safe API, etc.). Highest assurance.
[feed] sourced from a feed provider's published assessment via API or open repo. Re-verifiable at the source URL.
[curated] manually researched and added by a human curator via PR. Logged with editor and timestamp.
[self-reported] provided by the protocol team directly. Treated as a claim, not a fact.

Governance data is fetched live from the Safe API where possible, so multisig thresholds and signer counts reflect current on-chain state rather than stale YAML entries. Where the Safe API fetch fails, the static fallback is shown with an explicit degrade notice.

The feed registry

Full list of all 17 providers Relatum aggregates from. Each feed has its own methodology and focus. We do not rank them. v1 covers the 14 providers named in the RFP feed registry plus 3 proposed additions (Chronicle, Accountable for verifiability; Stablewatch for the yield-bearing stablecoin axis) widening methodology diversity to 6 types: Rating, Dashboard, Monitoring, Research, Verification, Analysis.

Per-feed schema docs live alongside the registry at data/feeds/<feed>.yaml — entity model, scoring dimensions, anchor examples, and known coverage gaps for each provider. Twelve of the seventeen are committed at v0.1; the remaining five (pigi.finance, Zyfai Risk, LlamaRisk, Chronicle, Accountable) land as feeds are onboarded.

How to contribute

All Relatum data is open source under AGPL-3.0. Anyone can propose:

Corrections

Open a GitHub issue describing the inaccuracy, or drop a JSON patch in data/overlays/ and open a PR. Triple-level supersedence via rfp:supersededBy preserves history.

New protocols

Add a YAML entry under data/protocols/. The build pipeline mechanically generates the per-protocol Turtle stubs; reviewers fill in feed coverage incrementally.

New feeds

Drop a <feed>.yaml schema doc in data/feeds/ describing entity model, scales, and known gaps; add a .ttl file with the first claims; register the feed URI in vocab.ttl. No graph-schema change; the matrix extends automatically.

M2 deliverable: at least one round of external corrections accepted and merged, with the workflow proven end-to-end. Charter governs the no-composite-scoring constraint.
